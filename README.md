## Features

- **Menu bar countdown** — see your next event and a live relative timer (`in 12m`, `in 1h 5m`, or `now`)
- **Day navigation** — browse past and future days with quick prev/next controls
- **Calendar filtering** — pick which calendars to monitor in Settings
- **Launch at Login** — start automatically so you never miss a beat
- **Self-updates** — direct installs can update in place with Sparkle once the release feed is configured

## Requirements

- macOS 14.0+
- Xcode 16+

## Build and Run

```bash
xcodebuild -project UpNext.xcodeproj -scheme UpNext -configuration Debug -derivedDataPath /tmp/UpNextDerived build
open /tmp/UpNextDerived/Build/Products/Debug/UpNext.app
```

Or open the project in Xcode:

```bash
open UpNext.xcodeproj
```

## App Updates

Direct-release and Homebrew Cask builds self-update with Sparkle. The feed is hosted as the `appcast.xml` asset on the latest GitHub release. A distributable build also requires:

- Developer ID Application signing and Apple notarization
- `SPARKLE_PUBLIC_ED_KEY` set to the public EdDSA key generated by Sparkle
- the matching private EdDSA key kept outside the repository

If the public key is unset or invalid, the updater stays disabled and the Settings pane explains what is missing. UpNext requires signed feeds and verifies update archives before extracting them.

Resolve the package once to make Sparkle's official release tools available, then generate the key in your login Keychain and securely back it up:

```bash
swift package resolve
.build/artifacts/sparkle/Sparkle/bin/generate_keys --account upnext
.build/artifacts/sparkle/Sparkle/bin/generate_keys --account upnext \
  -x /secure/path/upnext-sparkle-private-key.txt
```

For non-interactive notarization, create an App Store Connect API key in **Users and Access → Integrations**. Give it the Developer role, download its `AuthKey_<KEY_ID>.p8` file, and record the Key ID and Issuer ID. Store the API credentials as a reusable Keychain profile:

```bash
xcrun notarytool store-credentials UpNext \
  --key /secure/path/AuthKey_YOUR_KEY_ID.p8 \
  --key-id YOUR_KEY_ID \
  --issuer YOUR_ISSUER_ID \
  --validate
```

This does not use an Apple ID or app-specific password. Future releases can use `NOTARY_PROFILE=UpNext` without an interactive prompt. For an individual API key, omit `--issuer`.

Build, Developer ID-sign, notarize, staple, and package the universal app:

```bash
SPARKLE_PUBLIC_ED_KEY='YOUR_PUBLIC_KEY' \
NOTARY_PROFILE=UpNext \
./scripts/build-distribution.sh
```

Place release notes beside the generated archive using the same basename, for example `UpNext-1.0.0.md`. Preserve the previous release's `appcast.xml` in `artifacts/releases`, then generate the next feed with Sparkle's official tool:

```bash
./scripts/generate-appcast.sh \
  --archives-dir artifacts/releases \
  --download-url-prefix \
    https://github.com/srichand/UpNext/releases/download/v1.0.0/ \
  --keychain-account upnext
```

For CI, pass an exported private key using `--private-key-file -` on standard input instead of importing it into the runner Keychain.

Upload these flat assets to the `v1.0.0` GitHub release:

- `UpNext-1.0.0.zip` — shared by Sparkle and Homebrew
- `UpNext-1.0.0.md` — Sparkle release notes
- `appcast.xml` — signed update feed
- `upnext.rb` — generated Homebrew Cask definition

## Homebrew

The generated Cask declares `auto_updates true`, so Homebrew installs the notarized app and Sparkle keeps it current afterward. Publish `upnext.rb` at `Casks/upnext.rb` in a `pendyala/homebrew-tap` repository. Users can then install with:

```bash
brew install --cask pendyala/tap/upnext
```

Once UpNext meets Homebrew's notability requirements, the same Cask can be proposed to `Homebrew/homebrew-cask` for the shorter command:

```bash
brew install --cask upnext
```

For a local, non-distributable Release install, continue using:

```bash
./scripts/release.sh
```
